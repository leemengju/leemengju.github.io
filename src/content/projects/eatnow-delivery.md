## 專案概覽

**01｜專案目標**:本專案為資展國際全端養成班 PHPD09 的期中小專題,目的在模擬真實的設計師與工程師合作模式,並運用課堂所學進行實際應用與整合。以「外送平台」為主題,針對都市上班族這一核心 Persona,進行專案的創意發想與功能開發,最終實現具有基本可用性的產品原型(MVP)。

**02｜角色與產出**:在這個專案中,我擔任 UX/UI 設計師與前端開發者,負責從產品構思到設計與開發的完整流程。首先進行產品發想,確立目標用戶並制定設計需求,接著進行 UI 設計,確保介面清晰易用;最後將設計稿轉化為具備核心功能的互動網站,實現高還原度的開發成果。

**03｜專案挑戰**:由於我在專案開始前並無任何程式開發經驗,整個開發過程可謂是「摸著石頭過河」。雖然 UI 設計與前端開發在目標上存在部分重疊(如實現直觀的操作體驗),但兩者的實現邏輯截然不同,這種差異在執行時帶來了極大的挑戰;加上程式開發的學習曲線陡峭,使得進度安排與排程充滿不確定性。

![開發任務排程:從介面 Redesign、切版到互動功能的完整時程](/legacy/project4/media27.png)

**04｜專案成果**:儘管過程充滿挑戰,最終成功超前達成預期成果。頁面設計與開發的還原度超過 90%,並完成構想階段的 UI Flow 和 Prototype;響應式設計(RWD)從 Figma 設計的單一斷點擴展至全載體範圍,實現更廣泛的設備兼容性。功能上,在原有規劃的基礎上新增會員管理、地址設定等互動性元素,進一步提升網站的使用體驗與功能完整性。

## 專案背景

隨著都市化進程加速,越來越多的上班族選擇透過外送平台來解決忙碌的生活與用餐需求。外送服務已不再只是便捷的選項,而是現代人生活中不可或缺的一部分。本專案目的在設計一個針對繁忙上班族的外送平台,提供快速、便捷且具有高品質餐飲選擇的服務,並藉由精心設計的網站提升用戶體驗,以滿足日益增長的需求。

**問題研究**:目前市場上的外送平台普遍存在幾項問題——繁瑣的操作流程、選擇過多導致用戶決策困難、無法快速找到符合需求的餐點、以及餐飲品質無法保證。用戶對平台的期望不僅是速度,還希望能在忙碌的生活中,快速獲得一個高品質且符合口味的選擇。

## 設計與開發執行

### 設計策略

針對上述痛點,網站設計圍繞「為繁忙的你,定義每一刻的美好滋味」這一 slogan 展開,強調簡潔流暢的介面,確保用戶能在最短時間內找到符合需求的餐點。透過精細的用戶體驗設計提升操作效率,同時在視覺與互動設計上追求清新、直觀,確保每一位用戶無論在何種設備上都能享受無縫的外送體驗。此外,平台聚焦於餐飲品質與快速配送,確保每一餐的美味與新鮮。

### 首頁

![首頁:桌機與行動裝置的響應式呈現](/legacy/project4/media12.png)

- **導航列(Nav-bar)**:自動帶入使用者定位與會員頭像。Menu 選單管理會員資訊與訂單內容,為外送員、企業主、供應商等不同角色(persona)提供快速連結;地址編輯器可快速切換使用者定位,並設定時間偏好與送達優先級。
- **標題區(Header)**:以清晰的功能分區強化視覺動線,幫助使用者快速掌握頁面資訊。
- **側邊欄(Sidebar)**:提供客製化篩選功能,包含餐點種類與星級篩選,解決用戶決策困難及快速找到目標餐點的需求。
- **優惠券與種類欄位**:設置左右滑動控制按鈕,適應不同裝置的螢幕尺寸差異,增強響應式設計(RWD)。
- **餐廳欄位**:分為「火速送達」、「專屬推薦」、「優惠方案」、「所有餐廳」四大類,搭配餐廳特色客製化標籤,幫助使用者快速分流至合適區域。
- **RWD 設計**:通過模組化設計實現桌機、筆電、平板與手機等多裝置的一致性使用體驗,使用者的視覺動線也會從水平移動變成垂直擴展。

### 餐廳頁面

![餐廳頁:桌機與行動裝置的響應式呈現](/legacy/project4/media13.png)

## 項目總結

在本專案的設計與開發執行中,我完成了從用戶登入、首頁設計到餐廳篩選、點餐與結帳流程的完整功能實現。系統透過 local storage 提升用戶登入的便捷性;首頁設計結合精準的用戶定位、導航欄與篩選功能,解決繁雜選擇與用戶痛點;同時,設計模組化的響應式網頁(RWD),保證跨裝置的使用體驗一致性。在餐廳與餐點頁面中,統一的設計語言和動態交互提升了用戶的視覺引導和操作效率,購物車和結帳功能則通過動態更新與即時報錯增強了使用流暢性。最終,外送頁面結合地圖與聯絡功能,進一步完善了平台的服務閉環。

## 反思與學習

- **Design System**:經過這次專案,我理解到不論是對設計師或工程師,Design System 對於系統的一致性與效率的提升都非常重要。
- **角色衝突與轉換**:設計與開發的邏輯差異需要更多的轉換——以設計背景做工程開發,反而可能限制產品功能的發展性,或因過度追求視覺而沒有深入了解程式運作邏輯,未來可嘗試更靈活的工作思路和流程。
- **需求與時間平衡**:在有限的時間內,需要有效篩選需求,集中資源開發用戶最需要的核心功能,不被其他人或開發中的突發奇想所影響。
- **檔案版本管理**:工程開發有一套撰寫邏輯,新手容易因為習慣而沒有分段程式碼,或是沒做好檔案區分管理,可能釀成之後多人協作的潛在隱患。
- **心態平衡**:不要因為看到別人的作品就開始懷疑和否定自己;永遠相信身邊的伙伴都是增強自身實力的後盾。無論如何,不內耗才是每次專案最關鍵的先決條件。

## 背景

公司規劃導入全新虛擬貨幣(Q 幣),但後台完全沒有任何現成頁面——沒有 DB schema、沒有前端畫面,一切要從零開始。既有的舊幣種報表近 50 份、各自獨立演化多年,Q 幣版本無法直接複製,每份都要先讀懂原邏輯才能判斷是「加欄位就好」還是「整個重寫」。加上產品規格未定,需要與管理層、運營、風管、客服等多部門逐張報表確認優先級,需求在確認過程中還會持續追加。

## 專案內容

與同事協作分工,在 8 週內從零交付全新貨幣系統後台,涵蓋報表、設定、點數管理、會員管理四大類,共 34 項工項上線,另有 15+ 項工項評估後確認不需要開發,有效收斂範疇。

## 專案挑戰

新舊幣種必須在同一後台介面並存,補幣、扣幣、統計等功能都要同時支援兩套邏輯,不能影響既有舊幣種邏輯與資料安全;同時公司有多條並行短線專案分支(8+ 條)同期開發,每次合版都要驗證 Q 幣邏輯沒被覆蓋。需求端本身也不穩定——規格確認過程中持續追加「新增需求」,已完成的功能常需二次改動;而近 50 份舊報表結構異質性極高,各有獨立 SQL 與前端結構,無法 1:1 複製,得逐份評估改寫深度。

## 個人貢獻

- 主導需求確認與工項分配:以 CSV 分工規劃表(優先級 × 可行性 × 負責人 × 完成進度)逐張報表與各部門對齊,每雙週向上層匯報進度,讓需求不確定期間的風險可視化。
- 個人獨立完成 16 項工項(購買免費遊戲報表、補幣管控中心、存摺紀錄、線上會員、道具卡等核心功能),提交 69 次 commits(Laravel 32 + Vue 37)。
- 設計四環境分支策略(測試 → 預發布 → 類正式 → 正式):上線前從預發布環境切出一條整合分支,把本版所有功能 commit 集中在該分支解衝突,驗證通過才逐級合回正式分支,讓正式環境在多條短線分支並行的情況下始終保持乾淨可上線。
- 建立 code 先行 / schema 後跟 的部署順序:DB 表尚未上測試環境前,在 Controller 內加「表是否存在」的判斷讓 code 先行合入不報錯,待表建好後的下一個 commit 才移除判斷、全面開啟功能,避免強依賴造成環境崩潰。
- 排除 MySQL 舊版對第一個 TIMESTAMP 欄位隱式加上 `ON UPDATE CURRENT_TIMESTAMP`、導致統計時間被覆蓋的坑;以及 Q 幣統計報表因非同步遊戲列表尚未 resolve 就先執行統計、導致結果為空的 Promise 順序問題。

## 專案結果與影響

準時在 8 週內交付全新貨幣系統後台,34 項工項完整上線;透過分支策略與雙週匯報機制,在多條並行短線分支與模糊需求的高壓環境下,仍確保交付品質與進度可視化。

## 關鍵技術決策與踩坑

**最痛的坑 1:舊版 MySQL 的隱式 TIMESTAMP ON UPDATE**

Q 幣專屬統計表建好後,一份時間相關報表的統計結果一直不對——第一個 TIMESTAMP 欄位的值在每次 UPDATE 後都被改成當下時間。一開始把矛頭指向 INSERT 邏輯,查了半天沒有結果;直到 `SHOW CREATE TABLE` 才看見真相:舊版 MySQL 會對表中第一個 `TIMESTAMP` 欄位隱式加上 `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`,即使 DDL 完全沒寫。舊幣種原表因欄位順序不同沒有踩到,所以無前例可循。教訓:舊版 MySQL 建完含 TIMESTAMP 的表,一律先驗證 schema。

```sql
-- 建完表後一律確認:第一個 TIMESTAMP 欄位是否被隱式加上 ON UPDATE
SHOW CREATE TABLE `<your_stats_table>`;
-- 若出現非預期的 ON UPDATE CURRENT_TIMESTAMP,手動修正
ALTER TABLE `<your_stats_table>`
  MODIFY `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP;
```

**最痛的坑 2:非同步遊戲列表尚未 resolve 就先統計**

Q 幣統計報表在 `mounted` 階段同時觸發「載入遊戲列表」與「執行統計查詢」,但遊戲列表是非同步 action,統計查詢沒等它完成就跑,導致篩選用的遊戲列表為空、統計結果錯誤。一開始誤判成箭頭函式作用域問題,改了兩次才確認不是根因。正解是把統計查詢接在列表 Promise resolve 之後執行。

```js
// 錯誤:兩者並行,統計時列表可能還沒好
this.getGameList()
this.searchStats()

// 修正:列表就緒後才統計
this.getGameList().then(() => this.searchStats())
```

**值得保留的工程實踐**

- 四環境分支策略(測試 → 預發布 → 類正式 → 正式):衝突集中在整合分支解,預發布與正式分支始終保持乾淨。在 8+ 條短線分支並行時,這個流程是正式環境不出事的關鍵。
- code 先行 / schema 後跟:以「表是否存在」的判斷解耦 code 與 DB 上線節奏,讓功能能安全地分兩步部署。

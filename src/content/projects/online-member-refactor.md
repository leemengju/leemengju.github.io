## 背景

舊版是一支累積五年補丁的單一頁面元件(712 行)加上一份輔助設定檔(232 行)。一支混亂的 API 混雜回傳彙總、遊戲清單、玩家清單三塊資料,任何一塊出問題都會影響整頁,也無法個別分頁或快取。更麻煩的是彙總值不是純數字,而是帶顏色的 HTML 字串(例如 `<span style="color:red">123</span>`),不同代版本的數字還被直接字串相加,前端完全無法對它做算術。四個部門各自有截然不同的痛點:執行長要在手機上一眼看完、運營總監要清楚看懂每個數字代表什麼、技術主管要能撐住上百款遊戲最多 11 個倍率的資料結構、風管要能看懂欄位意義而不用每次問工程師。

## 專案內容

把頁面拆成總表、遊戲明細、會員即時狀態三個獨立模塊,各自有專屬 API、獨立分頁邏輯與欄位定義,並讓總表的加總公式改成前端可即時調整,不用每次改需求都要動後端。撤回了原本規劃的自動刷新倒數功能。

## 專案挑戰

要在同一個版本裡一次滿足四個部門的異質需求,而且不能增加使用者的操作學習成本。最難纏的是彙總值「顏色字串相加變成一串奇怪長字串」的根因:運營總監反應總人數有時候正常、有時候變成開頭是正確數字的超長字串。一開始判斷成 API 偶發異常,查了後端 log 確認數字都是對的,才回頭發現是彙總值被包成 HTML 字串後,前端的 `+` 運算把兩段字串接在一起,不是相加。另一個難點是不同遊戲的倍率欄數量落差極大(1 到 11 個不等),舊框架只設計了四個倍率欄,完全塞不下。

## 個人貢獻

- 後端把單一混亂 API 拆成 3 支職責清晰的 API(總表彙總、遊戲明細清單、玩家即時狀態),各自可獨立分頁與快取,呼叫時機與錯誤處理互不干擾。
- 前端主檔從 712 行降為約 190 行的父層協調 + 7 個子模塊,各模塊只管自己的 template 與欄位定義,新需求加在對應子模塊即可,不影響其他分頁。
- 新增一個公式調整彈窗,讓使用者直接在 UI 上勾選「哪些欄位要加總」,不用改 code 就能調整總大廳人數的定義。
- 動態支援每款遊戲 1~11 種不等的倍率:統一輸出成四個倍率欄、不足補 0,並依 index 而非數值名稱映射(同一個 index 在不同遊戲對應到不同倍率值);超過四種倍率的桌牌類遊戲則在遊戲明細模式中靜默過濾,但不影響人數總覽模式。

## 專案結果與影響

執行長的手機友善排版、運營的清晰數字欄位、技術的結構重構、風管的明確欄位意義,四個部門同一版本同時上線,且不需要額外 UI 培訓。彙總值改成純數字後,前端可以直接做算術,不再有字串相加亂掉的問題;API 拆成 3 支後,單塊資料的問題不再拖垮整頁。

## 關鍵技術決策與踩坑

**最痛的坑 — 彙總值是「HTML 字串相加」而非數字相加**

彙總值原本被包成帶顏色標籤的 HTML 字串回傳,不同代版本的數字還被字串串接在一起。前端拿到後用 `+` 運算,結果不是相加而是接字串:

```js
// 症狀:後端回傳的是帶標籤的字串,不是數字
const a = '<span style="color:red">123</span>';
const b = '456';
a + b;            // "…123…456" → 畫面出現奇怪長字串,而非 579
Number(123) + Number(456); // 579 → 修正後:後端只回純數字,前端才做算術
```

因為後端 log 顯示的數字是對的,最初誤判成 API 偶發異常,繞了一圈才找到真正的根因在前端型別。修正方式是後端只回傳純數字、前端統一做加總。教訓很單純:彙總值只能是 `number`,絕不能帶任何 HTML 標籤。

**取捨 1 — 公式調整放前端而非後端**

否決的方案是讓後端算好「總人數」、前端只顯示結果。否決理由是執行長經常改「哪幾種人算進總人數」的定義,若放後端,每次調整都要改 code + 重新部署;放前端只是改設定,使用者自己在 UI 上勾就好。勾選狀態刷頁重置是刻意設計——執行長的調整多半是臨時性的,不需要永久化。

**取捨 2 — 遊戲明細不分頁,採全量前端排序**

否決的方案是分頁 + server-side sort(比照現有的人數總覽)。否決理由是:遊戲明細的資料量就是遊戲款數(排除桌牌後約 50~100 列),全量前端排序綽綽有餘;而分頁後的排序只能排當前頁,和使用者「我要排全部」的預期相反。因此改成後端一次回傳全量、前端 client-side 排序。

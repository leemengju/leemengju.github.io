## 背景

舊系統每個後台帳號只能對應一種角色,遇到複合需求(例如同時需要企劃 + 數值 + 組長的權限組合)就必須另外新建一個一次性角色,長期下來累積大量冗餘角色——同職位但功能組合不同的人,共用同一角色會權限過寬,分開管理成本又極高;而「全部權限」(groupNum=0)也無法與其他角色混用,邊界行為不明確。

## 專案內容

新增一張中間表讓一個帳號可以綁定多個角色,後端合併各角色的 rule 字串後,整份傳給前端,前端改為 checkbox 複選;同時保留既有 `DG` 欄位向下相容,不改動現有登入與權限驗證流程。

## 專案挑戰

Rule 字串合併需要手工解析格式(`key_suffix` 字元級 union),邊界情況不少——尤其「全部權限」groupNum=0 這個值,在 PHP 的 truthy 判斷裡很容易被誤傷:

```php
// 有坑:0 > 0 為 false,全部權限(groupNum=0)整個被濾掉
$groups = array_filter($input, fn($g) => $g > 0);

// 修正:改成 null/空字串判斷,讓合法的 0 通過
$groups = array_filter($input, fn($g) => $g !== null && $g !== '');
```

同樣的坑也出現在合併後的輸出:`permission_group.rule` 對「全部權限」存的是 `"all"`(不含底線),但合併演算法照 `key_suffix` 格式解析,會把它重組成 `"all_"`,前端驗證因為預期 `"all"` 而判斷失敗,需要額外加一段特殊處理。

## 個人貢獻

- 新建 `administrator_permission_groups` 關聯表(`sn`, `account`, `groupNum`, `unique(sn, groupNum)`),實作 `mergePermissions()` 演算法,將多個角色的 rule 字串依 key 做 suffix 字元級 union 合併。
- 以 Seeder 將所有帳號現有的 `DG` 值一次性遷移進新表,升級零停機、舊資料完整保留;`DG` 欄位持續同步寫入(取複選群組中的第一個),讓現有讀取 `DG` 的程式碼無感升級。
- 前端複選 UI 加入「全部權限只能單選、選了不能再加其他群組」的攔截邏輯。

## 專案結果與影響

帳號從只能對應單一角色,升級為可複選、不限數量的多角色組合;複合需求不再需要另建角色,前端也不用再自行解析 rule 格式,直接使用後端合併好的 `ruleMerged`。

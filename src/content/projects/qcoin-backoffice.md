## 背景

- **全新幣種,毫無基礎**:公司規劃導入全新虛擬貨幣(Q 幣),後台完全沒有任何現成頁面,需從 DB schema 設計到 Vue 前端全部新建。
- **近 50 份報表、結構各異**:既有幣種的報表各自獨立演化多年,Q 幣版本無法直接複製,每份都需要評估可行性、重新開發或深度改寫。
- **需求極度模糊**:產品規格未定,需與各部門(管理層、運營、風管、客服…)逐張報表確認優先級與細節規格,需求會隨確認過程不斷追加。
- **兩個月的死線**:目標是在極短的兩個月內擴大潛在玩家群,時程壓縮,不允許長期研究。
- **與既有幣種系統的相容性**:新系統必須與原有幣種模組並存,補幣、扣幣、統計等功能需在同一後台介面中支援兩套幣種。
- **本線與多條短線分支的合版衝突**:多條短線專案同期並行,合版時需頻繁解衝突。

## 目標

與同事兩人協作,在 2026-03-10 ～ 2026-05-05 內交付 Q 幣後台的後端與前端,優先完成 Priority 1～2 的核心報表與管理功能,並建立持續可追蹤的工項分配機制,不投入評估為「不需要」的功能。

- **全新 DB 建表與相容設計** — 新建 Q 幣專屬資料表,同時確保共用的玩家資產表向下相容,既有幣種邏輯不受影響。

## 解法與架構

| 模組 | 做法 |
|---|---|
| **需求管理** | 以 CSV 表格列出所有候選功能(種類 / 功能 / 優先級 / 可行性 / 負責人 / 完成進度),逐張與各部門確認後鎖定範疇。 |

| **DB 相容設計** | 新建 Q 幣專屬統計表;共用的玩家資產表以新增欄位方式擴充,不修改既有幣種欄位,避免相容性風險。 |
| **母卡/子卡設計** | Q 幣卡後台只需儲存母卡資料(一組母卡代碼),client 端自行解析對應圖片;DB 不需存子卡,降低後台複雜度。 |

## 困難點

- **需求不穩定**:規格確認過程中持續追加「新增需求」(10+ 筆),導致已完成功能需二次改動。
- **各報表結構異質性高**:既有幣種報表各有獨立 SQL 邏輯與前端結構,Q 幣版本無法 1:1 複製,需逐份評估改寫深度。
- **跨幣種 UI 共存**:同一後台頁面需同時支援兩套幣種(如補幣管控中心新增三欄位、最大值邏輯各異),UI 改動容易誤觸原有邏輯。
- **MySQL 版本差異**:舊版 MySQL 對第一個 TIMESTAMP 欄位有隱式 `ON UPDATE CURRENT_TIMESTAMP` 行為,建表後需手動修改 schema,既有幣種舊表沒有這個問題。
- **短線分支頻繁合版**:並行 8+ 條短線分支,每次合版都需驗證 Q 幣邏輯未被覆蓋。

## 最痛的坑

**最痛的坑 1:舊版 MySQL 的隱式 TIMESTAMP ON UPDATE(購買免費遊戲報表)**

- **症狀**:Q 幣專屬統計表建好後,第一個 TIMESTAMP 欄位的資料在每次 UPDATE 後時間戳記會被自動覆蓋,統計結果異常。
- **誤判**:初步以為是 INSERT 邏輯寫錯。
- **根因**:舊版 MySQL 對表中第一個 `TIMESTAMP` 欄位預設加上 `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`,即使 DDL 未明確指定。既有幣種原表因版本或欄位順序不同而未觸發,所以無前例可循。
- **解法**:建完表後手動執行 `ALTER TABLE` 修改 schema,去掉隱式 ON UPDATE 行為;此後在舊版 MySQL 新建含 TIMESTAMP 的表,一律先以 `SHOW CREATE TABLE` 確認有無隱式 ON UPDATE,才不會再被同一個問題絆住。

```sql
-- 建完表後一律確認:第一個 TIMESTAMP 欄位是否被隱式加上 ON UPDATE
SHOW CREATE TABLE `<your_stats_table>`;
-- 若出現非預期的 ON UPDATE CURRENT_TIMESTAMP,手動修正
ALTER TABLE `<your_stats_table>`
  MODIFY `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP;
```

**最痛的坑 2:非同步遊戲列表尚未 resolve 就先統計(Q 幣統計報表)**

- **症狀**:統計查詢被呼叫時,篩選用的遊戲列表尚未 resolve,導致資料為空、統計結果異常。
- **誤判**:以為是箭頭函式作用域問題,改了兩次箭頭函式才確認不是根因。
- **根因**:`mounted` 階段載入遊戲列表是非同步的 Vuex action,但統計查詢沒有等待其完成就執行。
- **解法**:改為在遊戲列表的 Promise resolve 之後才執行統計查詢,確保資料就緒後再統計。

```js
// 錯誤:兩者並行,統計時列表可能還沒好
this.getGameList()
this.searchStats()

// 修正:列表就緒後才統計
this.getGameList().then(() => this.searchStats())
```

## 附錄

**可複用經驗**:舊版 MySQL 建含 TIMESTAMP 欄位的表,建表後一律以 `SHOW CREATE TABLE` 確認無隱式 ON UPDATE;依賴非同步 Vuex action 資料的統計查詢,必須等該 action 完成後才執行。

## 小結

本次是 PM 與工程師雙身分並行的開發經歷:需求極度模糊、開發體量龐大、時間軸極緊,且需多部門配合協調。這個專案的價值不在單點技術突破,而在如何在混亂中穩定推進——回頭看 commit,工程師視角有幾件事值得記下來:

**1. 讀懂異質舊 code、判斷擴充深度**

近 50 份既有幣種報表各自演化、結構不同,每份都需要先讀懂原版邏輯,才能判斷 Q 幣版本是「加欄位就好」還是「需要重寫」。例如線上會員報表,需先讀懂既有的遊戲列表篩選邏輯,才能在不破壞既有幣種的前提下新增 Q 幣押注額與專屬機台人數;Q 幣存摺則是先理解既有存摺的 Controller 結構後,才從頭建立對應的 Controller 與資料模型。

**2. 環境特定 bug 的排除法:從現象回推根因**

先發現症狀(報表數字錯),再追根因(schema 行為),而不是從 schema 開始查。購買免費遊戲報表的統計時間不對,初判是 INSERT 邏輯或統計 SQL 寫錯;回頭檢查 schema 才發現舊版 MySQL 對第一個 TIMESTAMP 欄位預設加上 `ON UPDATE CURRENT_TIMESTAMP`,每次 UPDATE 都會覆蓋時間戳記。既有幣種舊表因版本或欄位順序不同未觸發,所以沒有前例可循。

**3. 四環境 Git 分支策略與上線流程**

專案有四條主分支對應四個環境:測試 → 預發布 → 類正式 → 正式。上線前兩週的實際流程:

1. 從預發布環境切出一條整合分支。
2. 把本次版本所有功能 commit 合進去,在這條分支上集中解衝突。
3. rebase 預發布分支,再將整合分支 merge 回預發布。
4. 驗證預發布環境無誤後,才逐級合入正式。

這個流程的價值在於:衝突集中在整合分支解,預發布與正式始終保持乾淨。在 8+ 條短線分支並行的情況下,沒有這個流程很難保證正式環境不出問題。

**4. 了解系統側效應:一個看似局部的改動影響全局**

對某個功能「先擋起來」,背後往往是真實測試後發現的系統依賴,而不是直覺保守。例如 Q 幣道具卡先擋掉刪除按鈕——測試時實際刪除後,發現伺服器整個無法重開(找不到該道具卡的 item 定義),影響範圍遠超預期。不擋不知道,擋了才知道刪除有全局副作用。

**5. Code 與 DB schema 的部署順序管理**

DB 表還沒上測試環境,但 code 要先進版本控制。如果 code 直接讀一張不存在的表,測試環境會立刻報錯。做法是在 Controller 裡加入「檢視 table 有無」的判斷,讓 code 可以先合入不報錯;待表建好後的下一個 commit 才正式移除判斷、全面開啟功能。這是 code 先行 / schema 後跟 的部署策略,避免強依賴造成環境崩潰。

> 這個專案帶來了過去主管分配式開發模式難以獲得的三件事:
> 1. **使用者需求訪談**的實踐 — 逐張報表與各部門確認,真正了解後台系統的實際使用情況。
> 2. **長線開發的合版實戰** — 多人多分支的真實場景,親身處理長線與多條短線分支的衝突解耦。
> 3. **範疇管理的實踐** — 主動與各部門確認並剃除 15+ 項不需要的功能,不是被動接受全部需求。

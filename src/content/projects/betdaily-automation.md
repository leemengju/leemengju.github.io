## 背景

> [!IMPORTANT]
> **核心痛點:每天花 20 分鐘填 Excel,填完還不知道有沒有填對。**

這份日報表牽動三個層級的使用者:

- **風管(執行層)**:每日凌晨手動查前一日各遊戲押注額、填 Excel、逐款換算,20 分鐘的重複勞動;自動化後從「數據搬運工」回歸本職的分析與回報。
- **運營長(策略層)**:依賴風管整理的 Excel 制定策略,但 Excel 欄位固定、遊戲無法比較排序,數字也無從驗證;改版後可直接看折線圖做多遊戲趨勢比對,匯出順序依上架時間對齊。
- **執行長(審閱層)**:定期審閱報表,過去只能接受「不知道有沒有填對」的數字;資料改由系統自動從 DB 統計後,誤填問題從源頭消除,審閱的數字是可信的。

原始痛點:

- **每日重複勞動**:每日凌晨人工查前一日各遊戲押注額並填入 Excel,一款一款換算,每日耗時 20 分鐘。
- **無驗證機制**:依「上線後第幾天」欄位人工填寫,誤填或漏填無法察覺,數字可信度低。
- **查詢與比較困難**:Excel 欄位固定,多款遊戲無法調動順序,以遊戲為基準的橫向比對很吃力。

上線後又冒出的新痛點:

- **重跑需求**:有遊戲重新上架,主管要求從當天起重算,但日報表仍從舊上線日查,無法修正。
- **無法記憶查詢組合**:每次打開都要重新勾選遊戲,無法保存常用查詢組合。
- **匯出排序問題**:匯出後欄位排序不直觀,應最舊遊戲在左、最新遊戲在右。

## 目標

- 以 Kernel 定時排程取代人工填寫,解決每日重複勞動與無驗證機制。
- 以 v-charts 折線圖 + vuedraggable 可拖拉表格,解決查詢比較困難與匯出排序。
- 以重跑功能(UI + CLI)解決重新上架後的重算需求。
- 以範本管理解決查詢組合無法記憶的問題。

## 成果亮點

1. **每日人工作業清零**:Kernel 每日 01:00 自動執行統計 pipeline,按上線日起逐日累積 60 天押注額,無需人工介入。
2. **數字可驗證**:資料直接來源遊戲明細資料庫,以 DB 統計取代 Excel 人工填寫,誤填問題從源頭消滅。
3. **多遊戲視覺化分析**:v-charts 折線圖顯示各遊戲 60 天趨勢,vuedraggable 允許拖拉表格欄位任意重排,輔助橫向比對決策。
4. **重跑修復任意遊戲**:UI 選遊戲 + 上線日期即可觸發重跑,重跑操作寫入後台操作 log,歷史記錄可審計。
5. **範本記憶查詢組合**:使用者可建立遊戲範本(含範本名稱 + 遊戲清單),主頁以 el-select 一鍵套用,不需重新勾選。

## 解法與架構

### 後端 Pipeline(每日 Kernel 01:00)

```mermaid
flowchart LR
  K["Kernel 排程<br/>每日 01:00"] --> A["補入新上架遊戲"]
  A --> B["確認各遊戲上線日"]
  B --> C["統計昨日押注額"]
  C --> D["shift 60 天陣列<br/>寫入記錄表"]
```

| 步驟 | 說明 |
|------|------|
| 1 補新遊戲 | 從遊戲主檔清單補入尚未建檔的新遊戲到上線日期標記表 |
| 2 確認上線日 | 查遊戲資料庫(open=1)+ 各遊戲第一筆建立時間,確認上線日 |
| 3 統計押注額 | 對狀態為 finished 的遊戲統計昨日押注額,shift 進 60 天陣列 |
| 4 串接 | 依序串接 1 → 2 → 3 成單一 pipeline |

> [!NOTE]
> 目前涵蓋 40 款遊戲,pipeline 單次執行約 10 秒。Kernel 失敗時有 Slack 通知;若資料異常,UI 一鍵重跑即可補算。

### 資料表

| 表 | 用途 |
|----|------|
| 押注額記錄表 | 每款遊戲一筆,以 JSON 存 60 天押注額浮點陣列(單位億,floor 到小數第 1 位) |
| 上線日期標記表 | 記錄每款遊戲的上線日與統計狀態(notYet / finished) |
| 範本設定表 | 範本名稱 + 逗號分隔的遊戲 id 清單 |

### 前端元件與視覺化

- **主頁**:遊戲選擇器 + 範本 el-select + 折線圖(v-charts)+ 可拖拉表格(vuedraggable)+ el-table + 匯出。折線圖自動呈現 60 天趨勢,少量遊戲對照時 Tooltip 精準顯示某天各遊戲的押注額;表格天數固定在最左、遊戲欄位可自由拖動,並以 chip 列顯示目前欄位順序。
- **重跑**:遊戲 el-select + 上線日 el-date-picker + 確認 dialog + 操作記錄表格。
- **範本**:範本清單管理 dialog + 範本新增/編輯 dialog(過濾停用與非表列遊戲)。

多版本 log 相容:重跑操作記錄跨越了一次欄位調整,read 端以 `gameName ?? gameId` 之類的 fallback 顯示,舊紀錄才不會開天窗。

## 最痛的坑

### 坑 1:`update()` 回傳 0 筆被誤判成「找不到資料」

範本編輯時,如果使用者送出的內容跟現有資料完全相同,API 會回傳失敗,使用者困惑「我明明沒改什麼,為什麼不能存」。根因是原本程式碼把 update 的 affected rows = 0 當成「找不到資料」而回報失敗;但 Laravel 的 `update()` 在資料無變動時本來就回傳 0 affected rows —— 這只代表「什麼都沒變」,不代表「找不到」。修法直接移除這個判斷:update 沒拋例外就視為成功。也正因為 affected rows = 0 等於 nothing changed 而非 not found,任何允許送出同值的設定型編輯都不該用 affected rows 判斷成敗。

```php
// 錯誤:允許送同值的設定頁,不能用 affected rows 判斷成功
$affected = DB::table('bet_report_template')
    ->where('id', $id)
    ->update($data);
if ($affected === 0) {
    return $this->fail('NO_RESULTS'); // 送同值時誤判!
}

// 正確:update() 未拋例外即成功;affected=0 只代表 nothing changed
DB::table('bet_report_template')->where('id', $id)->update($data);
return $this->success();
```

### 坑 2:el-select 清空事件把已選遊戲清光

主頁的範本 el-select 加了 `clearable`,使用者點叉叉清空後,先前手動勾選的遊戲整批消失。根因是 `@change` 在清空時也會觸發,此時傳入的 value 是 `null`,下游用它去 `find()` 範本得到 undefined,再 parse 就變成空陣列,把既有選擇覆蓋掉了。修法是在 handler 開頭對 `null` 直接 early return —— `clearable` + `@change` 的元件,清空會以 `null` 觸發同一個事件,handler 一定要在開頭處理 `null`,否則下游的 find/parse 會反過來清掉使用者既有的資料。

```js
function onTemplateChange (id) {
  if (id == null) return          // 清空事件 → 不動既有選擇
  const tpl = templateOptions.find(t => t.id === id)
  games = parseSelectGameList(tpl.selectGameList)
}
```

## 關鍵取捨

1. **老遊戲用前端注入,而非改 DB schema**:有幾款早期上線的遊戲不在標準遊戲清單邏輯內。與其為這幾款去改資料表結構或污染統計 pipeline,選擇在前端以常數定義並透過遊戲選擇器的參數注入。代價是這幾個 id 硬編碼在前端、未來若要新增需改 code,但這批老遊戲沒有新增計畫,以最小侵入換取 pipeline 邏輯乾淨。
2. **排程 class 命名不動,只重整 controller 命名空間**:功能長大後,原本以「60 天押注額」命名的控制器職責已擴及範本、重跑,於是把 controller 收進統一命名空間與子資料夾;但排程 job 的 class 名稱維持原樣 —— 因為 class 名稱不影響路由與使用者,改它反而有中斷既有每日排程的風險,不值得。

## 量化成效

| 項目 | 之前 | 之後 |
|------|------|------|
| 每日人工作業時間 | 一款遊戲 60 天週期累積 20 小時 | 0 小時(Kernel 全自動) |
| 數字可驗證性 | 人工填寫、誤填無法察覺 | 直接從 DB 明細統計,可審計 |
| 遊戲順序可調 | Excel 欄固定 | vuedraggable 任意拖拉 |
| 重跑上架日期修復 | 無法做到 | UI 一鍵重跑 + 後台操作 log 審計 |
| 記憶查詢組合 | 每次重勾 | 範本一鍵套用 |
| 匯出排序 | 無規則 | 依上架時間排序(最舊在左)|
| 統計遊戲規模 | — | 40 款遊戲,pipeline 單次 ~10 秒 |
| 解決痛點數 | 0/6 | 6/6 |

## 未來規劃

- **超過 60 天的遊戲**:押注額記錄表固定 60 天陣列,60 天後繼續 shift 移除最早一天;若未來有「查所有歷史」需求,需改成分區表或另開 history 表。
- **重跑操作記錄分頁**:目前每次開啟 dialog 載入第一頁;若重跑頻繁,歷史列表可能過長,之後可考慮增加日期篩選。

## 檔案結構整理

以功能長大後的一次重整為代表:原本是單一控制器 + 扁平頁面,重整後收進模組命名空間與子資料夾。

**整理前(flat)**

```
app/Http/Controllers/
  報表控制器.php            # 命名沿用「60 天押注額」,職責單一

src/page/report/
  報表主頁.vue              # 扁平頁面,無子元件分離
```

**整理後(模組命名空間 + 子資料夾)**

```
app/Http/Controllers/betDaily/
  報表控制器.php            # 含全部原方法 + 重跑 / 後台操作 log 讀取
  範本設定控制器.php        # 範本 CRUD

src/page/report/betDaily/
  報表主頁.vue              # 主頁(工具列左右分區、範本選擇)
  rerun/
    重跑對話框.vue          # 重跑 Dialog + 確認
    操作記錄表格.vue        # 後台操作 log 分頁表格
  template/
    範本管理對話框.vue      # 範本清單管理
    範本編輯對話框.vue      # 範本新增 / 編輯
    範本API.js              # 範本 API + 解析
```

> [!NOTE]
> 模組只要超過一個 Vue/JS 檔案,一律用子資料夾包起來(rerun/ 2 個、template/ 3 個),避免頁面目錄扁平後難以維護。

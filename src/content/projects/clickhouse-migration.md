## 背景

> [!IMPORTANT]
> **核心痛點:查詢「慢」+ 數字「不可信」,兩者都是痛**。

- **數字常和實際對不上**:舊報表的數字來自「後台統計表」,統計表會**掉 log、得人工重跑**。
- **多部門長期反覆回報「數字不對」**:每次都要人工追、重算、補。
- **衍生出一批 workaround 統計表**:因為直接掃 MySQL 明細太慢,才預先算好一批後台統計表頂著;遷到 ClickHouse 即時聚合後不再需要。
  - ⚠️ 界定清楚:**因「需要重置」或「需要定時紀錄」而跑的統計不算**在消除範圍內 —— 那些有其本身的業務理由,不是為了繞過慢查詢。
- **跨月份查詢慢**:原本跨月掃明細幾十秒。

## 目標

- 讓報表數字直接來自源頭資料(即時聚合),從根解掉「數字不對」與「報表慢」。

## 成果亮點

1. **數字可信,不再被反覆質疑**:改為直接對源頭(明細 / raw)即時聚合 → 跨部門「數字不對」的回報從源頭消失。
2. **查得到即時資料**:過去只能查「已寫好的統計」,現在含當下即時(今天走 raw、過去完整日走聚合表)。
3. **查詢大幅提速**:整月 **約 104s → 約 5s**(同範圍實測,~21x)。
4. **維護成本大降**:不再養那批「寫統計」的排程與補救邏輯(重跑、補資料、處理掉 log)。
5. **程式碼集中、好接手**:玩家 / 專屬機台 / 依遊戲分表的子報表,從四散的單一巨型檔案與多個 controller,整併成領域層 + 一域一檔(見[檔案結構整理](#檔案結構整理))。

## 解法與架構

依子報表來源型態,分兩種模式:

| 子報表 | 來源 | 模式 | 關鍵概念 |
|--------|------|------|----------|
| 總勝分(玩家) | 單一大型明細表 | **MV + 聚合表** | Materialized View(insert 時自動聚合)→ AggregatingMergeTree 聚合表(查報表用) |
| 依遊戲分表的子報表 | 數十張同結構分表 | **合併視圖 + 聚合表 + Kernel 排程 pull** | 合併視圖(命名規則自動含括新遊戲)→ 排程增量同步(everyMinute,浮水印機制)→ 聚合表 |

- **計分一律走 aggregate state**:sum / 去重合併函式,各批 partial state 自動 merge。
- **「今天」一律走 raw**:聚合表只到上次同步浮水印、未必完整;過去完整日讀聚合表,跨今天則「過去日聚合狀態 ∪ 今天 raw 狀態後 merge」(人數跨段去重正確)。

## 上線策略

- **雙軌並行**:切換 ClickHouse 時 MySQL **沒有停**,兩邊同時提供資料,確認 CH 正確後才切換報表來源,零停機。
- **維運窗口**:在固定每周維運時間上線,降低對用戶的影響。
- **Rollback**:程式層 revert commit 即可回到 MySQL 路徑,無需額外機制。

## 資料驗證

- **方法**:取同一個過去完整日,逐遊戲比對三邊各欄(押注額 / 輸贏 / jackpot / 局數 / 人數):**MySQL 明細**、**CH 明細**(人數用精確去重取值對拍)、**CH 聚合日表**(report 實際讀的)。
- **結果(37 遊戲 × 5 欄 = 185 格實測)**:
  - 列數:MySQL = CH 明細,**差 0**。
  - **CH 明細 vs MySQL:各欄完全一致**(含精確去重人數)→ ETL 入 CH 無誤。
  - **CH 聚合表 vs MySQL:金額 / 局數完全一致**;人數(近似去重)此日**誤差 0%**(HLL 小基數下即精確,容許 ~千分之一)→ MV/聚合邏輯無誤。
- **流程**:QA 雙軌比對通過後才切正式機。

## 異常處理與維運規劃

- **同步狀態表**:每次排程執行後記錄浮水印與狀態,可直接查表確認 sync 進度與最後成功時間。
- **錯誤告警**:排程失敗(例如正式機無法連上 ClickHouse)會自動送出錯誤通知工程師,不會靜默失敗。
- **Knowledge Transfer**:專案架構與操作邏輯已寫入內部技術文件,全團隊可查。

## 困難點

1. **AggregateFunction 欄不可直接 SELECT**:金額 / 人數等是聚合狀態,直接 select 會報錯,必須用對應的 merge 函式讀取。
2. **playerCount 不重複人數不能跨段相加**:相加會破壞去重。解法是讓日表的去重狀態與明細用**完全相同的篩選條件**產生,才能跨日/跨來源 merge 而去重不壞。改「日表狀態 + 只掃零頭日 raw」後,兩類機台分別從 24s、9s 降到 2.4s、次秒級,值 0 誤差。
3. **時區**:CH 伺服器底層是 **UTC**,靠連線層在請求帶入正確時區蓋上去。日期欄位是時區無關的 Date 型別,時間欄位比較則需轉換到當地時區。
4. **新遊戲自動納入**:分遊戲表會一直長新的,不想每加一款就補 MV/重佈。

## 最痛的坑

**curl_multi + libcurl < 7.20.0(正式機間歇全 0)**

- **症狀**:正式機間歇性「報表全 0 / 缺遊戲」,本地與 QA 打同一個 ClickHouse 卻都正常。log 顯示批次查詢裡的項目回傳空回應(`http=0, errno=0, body=""`),一次掉幾款、上午下午都有。
- **誤判過的方向(已排除)**:時區、連到不同節點/落後 replica、資料缺失 —— 全錯。靠「失敗回 null 被跳過 ≠ 真的 0」這個程式語意,以及 log 的空回應 signature,才鎖定是**連線層**。
- **根因**:併發查詢(curl_multi)的驅動迴圈舊條件寫成 `while ($running && $status === CURLM_OK)`,而這個寫法在 **libcurl < 7.20.0** 會爆。
  - **舊版規則(libcurl < 7.20.0)**:遇到「不必等 I/O、可以立刻往下推」的內部步驟(例如連線剛建立、內部狀態轉換)時,`curl_multi_exec` 會回傳 `CURLM_CALL_MULTI_PERFORM`(值 = **-1**,非 0),用意是「馬上再呼叫我一次,別去 select 等」。它**不代表某項成功/完成**,只是要你立刻再 exec 把這一串能立刻做的步驟消化掉;直到碰到要等 socket 才回 `CURLM_OK`。
  - **新版規則(libcurl ≥ 7.20.0)**:內部狀態機重寫,這些「能立刻做」的步驟改由函式庫自己一口氣做完,對外**永不再回** `CURLM_CALL_MULTI_PERFORM`,永遠只回 `CURLM_OK`(或錯誤)。
  - **出事點**:舊版第一次 exec 若回 `-1`,`$status === CURLM_OK`(=0) 判 false,外層 `while` 一次都進不去就**提前退出**,transfer 還沒跑完 → 還在飛的 handle 拿到空回應。(注意:`CURLM_OK` ≠ 完成;判完成看 `$running` 歸 0 / `curl_multi_info_read` 讀到 `CURLMSG_DONE`。)
- **為什麼只有正式機**:版本分界正是 **libcurl 7.20.0**。正式機的 libcurl 是 **7.19.7**(< 7.20,觸發此 bug),而本地與 QA 的 libcurl ≥ 7.20,所以怎麼測都正常。
- **解法**:內層 `do { curl_multi_exec } while ($status === CURLM_CALL_MULTI_PERFORM)` 把它排乾,外層 `curl_multi_select` 回 -1 時 `usleep(1000)` 防 busy-loop,並改用 `curl_multi_info_read` 取真實 CURLcode(`curl_errno` 在 multi 介面下常回 0、不可靠)。同時加「整批全失敗丟例外」避免靜默回一堆 null,並讓程式層對舊版 libcurl 健壯,而不是去動正式機的系統庫。

```php
// 有問題的寫法:只認一種「繼續」狀態,漏接中繼狀態碼
while ($running && $status === CURLM_OK) {
    $status = curl_multi_exec($mh, $running);
}
// 修正:先把「請立刻再呼叫我」的中繼步驟排乾,再進入等待 I/O 的迴圈
do {
    $status = curl_multi_exec($mh, $running);
} while ($status === CURLM_CALL_MULTI_PERFORM);
```

## 關鍵取捨

1. **依遊戲分表的子報表為什麼用合併視圖 + 聚合表 + Kernel,而不像玩家用 MV?**
   - 玩家來源是**單一表** → MV 可直接掛、insert 時即時聚合。
   - 依遊戲分表的子報表跨**多張**分遊戲表:(a) **MV 不能掛 Merge 表**;(b) 每張表各掛一個 per-table MV 又**無法自動納入新遊戲**、維運負擔大。
   - 故改:自建合併視圖(命名規則自動含括新遊戲)+ 排程主動讀、以浮水印半開區間增量 INSERT(每列只處理一次,不重不漏,遲到資料也補得進)。
   - **代價**:非即時(資料落後到上次同步時間,故「今天」一律走 raw)、且需要維護排程。**效益**:新遊戲上線時無需修改任何程式碼即可自動納入,且每筆資料保證只處理一次(exactly-once)。

2. **之後提速為什麼用 batch(並行),而不是加更細粒度的中間表?**
   - 曾評估加一層只掃當前小時的中間表。但實測瓶頸是**「開表固定成本 × N 遊戲序列往返」,與掃描量無關**(掃 1 小時 ≈ 掃整天)→ 細粒度中間表幾乎省不到,已移除。
   - 改用併發把 N 條 per-game query **並行**送出,固定成本重疊 → **約 6.5x**(序列 3347ms → 並行 515ms),值與序列一致。
   - 也否決過「N 遊戲 UNION ALL 成一條大 SQL」:單一管線內部並行差反而更慢,且一張表缺失整批失敗。

3. **人數去重為什麼用近似演算法而不用精確去重?**
   - 精確去重取精確不重複人數,但要建完整 hash set → 記憶體高、狀態龐大、整段掃描慢,且大狀態不利跨日/跨來源 merge。
   - 近似演算法是 HLL 近似,**誤差約千分之一**;營運統計的人數**容許這個誤差**,換來的是**小而可合併的 AggregateFunction 狀態 + 大幅查詢提速** —— 這正是日表能存人數狀態、跨段合併去重的前提。
   - 實證:playerCount 從整段 raw 掃 **約 24.2s** 降到日表狀態 merge 後 **約 0.1s**。

## 未來規劃

- 把同套聚合表模式推到其餘仍走 MySQL 的報表 / 其他幣別。
- 原始分遊戲表數量與容量持續成長,合併視圖本身 0 bytes、實際容量在底表。若要設保留策略,只能逐張原始表設 TTL/按月 DROP PARTITION,且保留天數須遠大於排程可能停擺時間,否則聚合表會出現永久缺口。
- CH HTTP client 可評估連線重用 / 連線池、調整併發數。
- 監控排程浮水印健康度(現有錯誤告警 + 狀態表可查,可進一步加停擺超過 N 分鐘主動告警)。

> **技術棧汰換**:正式機部分系統函式庫版本較舊、已 EOL,應排機器遷移到受支援版本,而非在舊機就地抽換系統庫(牽動範圍大、風險高)。

## 附錄

**可複用經驗**:

- 新報表提速:先判斷來源是「單一明細表」(→ MV)還是「多張分遊戲表」(→ 合併視圖 + 排程 pull)。
- 效能瓶頸先量測是「掃描量」還是「開表固定成本」,再對症下藥(見[關鍵取捨](#關鍵取捨))。

**佐證 benchmark**——主查詢對照。兩次量測跑的是**同一段日期區間的同一支報表查詢**,唯一差別是「有沒有讀聚合表」:一次強制走原始明細逐筆聚合(等同遷移前的舊路徑),一次走遷移後的聚合表,藉此得到乾淨的 before/after:

| 量測 | 主查詢耗時 |
|------|--------|
| 走原始明細、不讀聚合表(遷移前路徑) | **104292.8ms(≈104.3s)** |
| 讀聚合表(遷移後路徑) | **4992.51ms(≈5.0s)** |

子段:單看「不重複人數」這一段,從「每次都掃整段原始明細」改成「讀日表預存的去重狀態再合併」的 before/after(同一天、僅比對人數計算那段,非整月):

## 檔案結構整理

總勝分傳承多代、資料四散,加上開發者邏輯多元、開發時間短,原本架構零散。以一次重整為代表:把總勝分從**巨型單一檔案**與**散落的多個 Controller**,抽進**領域層** + **查詢層一域一檔**。

**整理前(零散)**

```
app/
├─ Model/Game.php                    # 巨型檔,總勝分邏輯混雜其中
├─ Http/Controllers/WinScore/
│  ├─ WinScoreController.php         # 主要控制器
│  ├─ WinScoreStat.php
│  └─ CCoin/
│     ├─ CheckDuplicateData.php
│     └─ DeleteWinScoreDayData.php
└─ Model/ClickHouse/WinScore/
   ├─ WinScorePlayer.php              # 玩家(既有)
   └─ WinScoreBuyFG.php               # 買免遊(既有)
   (其他子報表尚未獨立)
```

**整理後(領域層 + 一域一檔)**

```
app/
├─ Model/Game.php                    # 大幅瘦身
├─ Domain/WinScore/                  # 新增領域層,集中總勝分商業邏輯
│  ├─ TotalWinReport.php             # 報表
│  ├─ TotalWinStatistic.php          # ← 原 WinScore/WinScoreStat.php
│  ├─ CheckDuplicateData.php         # ← 原 WinScore/CCoin/
│  └─ DeleteWinScoreDayData.php      # ← 原 WinScore/CCoin/
├─ Http/Controllers/WinScoreController.php  # 瘦身為純入口
└─ Model/ClickHouse/WinScore/        # ClickHouse 查詢:一個域一檔
   ├─ WinScorePlayer.php             # 玩家
   ├─ WinScoreBuyFG.php              # 買免遊
   └─ WinScorePersonalSeat.php       # 專屬機台
```

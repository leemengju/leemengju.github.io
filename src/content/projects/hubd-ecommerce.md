## 專案概覽

### 01|專案目標

本專案為資展國際全端養成班 PHPD09 的畢業大專題,採取五人協作模式,並使用 React、Vite、Tailwind 和 Shadcn UI 開發前端,Laravel (PHP) 作為後端框架,資料存儲於 MySQL,並使用 GitHub 進行程式碼版本管理,確保系統開發環境穩定,最終完成電商平台的客戶端與企業端功能開發。

### 02|角色與產出

在本專案中,我擔任全端開發者與專案經理,負責從產品構思、設計到開發的完整流程。我主導了產品發想與排程規劃,確定目標用戶並設計網站架構,同時進行 UI 設計與開發。與團隊一同建置 React 與 Laravel 環境,設計資料庫結構,並進行前後端 API 串接,最後完成專案彙整並通過簡報呈現專案成果。

### 03|專案挑戰

由於大多數成員缺乏實際開發經驗,且需要在學習新技術的同時進行專題開發及準備面試筆試審查,時間壓力巨大。專案過程中,我們幾乎每天開會,確認功能需求、同步進度、協作除錯及專案整合,對新手工程師的抗壓性與自主學習能力提出了極大挑戰。

### 04|專案成果

儘管面臨諸多挑戰,我們最終成功完成專案的 MVP,實現了客戶端與企業端系統的獨立運行。通過 API 串接,實現了訂單資訊、商品賣場同步更新、行銷活動、金流交易、會員管理及系統設定等功能,優化了多方利害關係人的操作體驗,並確保了系統的穩定運行。

## 專案背景

### 選題動機

本專案聚焦快時尚電商平台,主要回應市場需求與產業趨勢。根據 Statista,2024 年全球快時尚市場規模預估超過 1,330 億美元,且持續成長;線上購物已占全球服飾零售超過 30%,顯示電商為主要銷售渠道。然而,傳統平台在商品管理與買賣互動上仍有不足,尤其中小品牌缺乏高效工具。本專案提供選品頁與賣家 CMS,協助品牌提升營運效率與用戶體驗,創造更高價值。

### 專案說明

本專案為快時尚電商平台,買家可透過平台瀏覽精選時尚服飾與配件。賣家則可透過賣家內容管理系統(CMS)高效管理業務,包括商品上架、訂單處理、會員管理、金流結算及活動行銷,確保運營順暢。

### 技術流程圖:前端

這張圖展示了本次專案 React 應用的架構。首先,應用從 index.html 啟動,並通過 src/main.jsx 進入核心邏輯(src/App.jsx)。應用使用 src/routers/Routers.jsx 來處理路由,顯示不同的頁面組件。每個頁面組件使用共享的 UI 元件來構建視圖。這些視圖會與服務層(如 api.js、AuthService.js)交互,調用後端 API(Laravel API)來處理數據。應用的配置由 tailwind.config.js、vite.config.js 等文件管理,確保樣式和構建過程的正確性。最後,靜態資源(如圖片和工具文件)被用來支持應用的各種功能。整個架構協同運作,確保了從前端到後端的順暢數據流與頁面渲染。

![前端 React 應用架構流程圖](/legacy/project5/media19.png)

### 技術流程圖:後端

這張圖展示本次專案 Laravel 的架構流程。當用戶發送 API 請求時,首先由 API 閘道(public/index.php)接收,並通過路由(routes/api.php)轉發至相應的控制器(app/Http/Controllers)。在這個過程中,請求會經過中介層(Middleware),進行身份驗證或其他處理。控制器會調用業務邏輯或模型(app/Models)來執行具體的業務邏輯,並與數據庫進行交互(MariaDB)。最終,靜態文件(如圖片)會從存儲(storage/app/public)中提取並返回給用戶。整個過程涉及了從配置、引導到數據存儲與文件處理的完整架構。

## 反思與學習

### 專案規劃

- 階段性規劃專案。設定每階段 MVP,並依照實際進度與資源適時調整,才能真正落實 Sprint 模式,避免瀑布式開發導致功能被迫刪減,也能減少前期的趕工壓力。
- 持續對齊使用者需求。設計功能時需要謹記專案規劃時的產品與使用者定位和 user story(情境),設計功能時才不會跑偏,開發出可能會用到但不那麼符合邏輯的功能。

### UI 設計

- UI 不能只想到美觀和易用性,同時要考量開發者體量和 API 串接難度。

### 開發執行

- 技術支援不夠到位,只在發生問題才給予幫助,沒有預先建立相關參考文件與基礎建設。
- 版本控制流程稍嫌複雜,且沒有根據實際情況調整,應該每完成一個小功能就合併到核心檔案中,避免內容被覆蓋或是最終專案合併困難。
- 由於缺乏 Code Review 機制,導致組員間程式碼邏輯不一致,進而增加後期維護的困難。例如:同樣的分頁功能(Pagination)與檔案匯出,其 UI 介面與實作方式皆存在差異,影響系統整體一致性與可讀性。

## 未來優化

### 第三方串接

- 客戶端能第三方登入(Line, Google),希望企業端能夠實現登入驗證、角色權限。如果有完善通知系統/實現信件通知,可依據角色開關不同的通知項目。
- 希望能完善金流和物流流程,串接綠界和 7-11 或全家超商地址 API。

### 資料庫

- 資料庫結構可透過 Migrate 與 Seeder 進行版本控管與資料初始化,若遇到錯誤可直接刪除重建以確保一致性;不過,此方式將使資料庫負責人的作業負擔相對較重。
- 資料庫建議由專責人員統一管理,避免多人同時修改導致資料可讀但無法更新,或更新後資料異常的情況。若需多人共同調整,建議先經由團隊討論並達成共識後再進行修改,以確保資料庫結構與邏輯一致性。
- 建議首頁的主打商品與下方商品卡片皆能透過後台系統進行管理。目前僅依商品 ID 順序自動抓取設定數量的商品,未來可考慮新增指定商品的管理功能,讓後台能靈活設定欲顯示的商品內容。

### Git 管理

- 建議由專責人員統一管理 Git 版本控制流程,其他組員可透過實際練習發送 Pull Request 並參與審核流程,藉此提升團隊對版本控管的熟悉度與整體協作的穩定性。

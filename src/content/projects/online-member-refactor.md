## 背景

舊版 `people_list.vue`(712 行)加上輔助設定檔(232 行)累積五年補丁,單一 API 混雜回傳彙總、遊戲清單、玩家清單,彙總值還用顏色字串回傳(例如 `<span style="color:red">123</span>`),不同代版本數字直接字串相加,前端完全無法算術。執行長、運營總監、技術主管、風管四個部門各自有截然不同的痛點:執行長要手機看得下、運營要清楚的數字欄位、技術主管要能撐住 116 種遊戲最多 11 個倍率的重構、風管要看得懂欄位意義不用問工程師。

## 專案內容

把頁面拆成總表、遊戲明細、會員即時狀態三個獨立模塊,各自有專屬 API、獨立分頁邏輯與欄位定義,並讓總表加總公式改成前端可即時調整,不用每次改需求都要動後端。

## 專案挑戰

要在同一個版本裡一次滿足四個部門的異質需求,且不能增加使用者的操作學習成本;同時要拆解舊版彙總值「顏色字串相加變成一串奇怪長字串」的根因——一開始還以為是 API 偶發異常,查了 server log 確認數字沒錯,才發現是前端 `+` 運算把兩段字串接在一起,不是相加。

## 個人貢獻

- 後端把單一混亂 API 拆成 3 支職責清晰的 API(總表彙總、遊戲明細清單、玩家即時狀態),各自可獨立分頁與快取。
- 前端主檔從 712 行降為約 190 行父層 + 7 個子模塊,新增 `SummaryFormulaDialog` 讓使用者在 UI 上勾選「哪些欄位要加總」,不用改 code 就能調整總大廳人數的定義。
- 動態支援每款遊戲 1~11 種不等的倍率,統一輸出 `gainRate1`~`gainRate4`、不足補 0,並依 index 而非數值名稱映射(不同遊戲同一 index 對應不同倍率值),棋牌遊戲則在遊戲明細模式中靜默過濾但不影響人數總覽模式。

## 專案結果與影響

執行長的手機友善排版、運營的清晰數字欄位、技術的結構重構、風管的明確欄位意義,同一版本同時上線,且不需要額外 UI 培訓;彙總值改成純數字後,前端可以直接做算術,不再有字串相加亂掉的問題。
